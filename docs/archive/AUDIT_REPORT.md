## Phase A — What the code actually is (and the docs delta)

kg-gen is a TypeScript CLI that turns a file tree into a knowledge graph: glob discovery → per-file reader (11 readers, first-match-wins factory) → chunking (LangChain recursive splitter; structural for JSON; turn-packing for transcripts) → optional corpus pre-pass (term frequency + one LLM call → glossary that becomes the **closed extraction vocabulary** under v5) → per-chunk LLM extraction (Zod-validated, closed enums for `entityType`/`relationType`, provenance + `validAt`/`createdAt` stamped deterministically) → optional inline grounding gate (keyword overlap) → per-chunk checkpoint (`--resume`) → 3-level merge (within-chunk → within-file → global; Jaro-Winkler + embedding cosine, digit-mismatch veto, cross-type bar 0.95) → optional post-merge transform pipeline (edge co-occurrence gate, embedding-clustering canonicalization with merge-log) → export via strategy (json / jsonl / mcp-jsonl / dot / kblam / lora / graphiti). Config is a single nested Zod schema (defaults live only there); DI is a custom async container; `schema`/`metrics`/`inspect-merges` subcommands exist beside the main run. A Next.js frontend drives runs over `--progress-ndjson`.

The June 5→10 sprint (Phases 0–4 + corpus/canon/merge-fix commits) genuinely landed: tests exist and pass, the `export` command is implemented, the double-count-on-rerun bug is fixed (prior output now feeds retrieval only — `DirectoryProcessor.ts:140-143`), `fileContent` is populated, KBLaM/LoRA/Graphiti exports exist, v5 prompts are the default.

## Top 5 by leverage

### 1. KG-01 — TextReader swallows every file; ASR is dead (crit; fix ≈ 20 lines + test)

`FileReader.canRead` (`FileReader.ts:52`) tests `ext.startsWith(e)`, and `TextReader.ts:12` includes `''` in its extension list — `'.mp3'.startsWith('')` is `true`, so TextReader claims **everything**. Registration order (`ContainerFactory.ts:279` Text before `:283` Audio) makes Whisper unreachable, and with no BinaryReader registered (it isn't even exported from `readers/index.ts`), unknown binaries are read as UTF-8 and shipped to the LLM as mojibake. Empirically reproduced with the exact production wiring: `song.mp3 → TextReader`, `video.mp4 → TextReader`, `x.sqlite → TextReader`. This has been latent since `419d5b8` — it survives because recent corpora (`.parakeet.txt`, chat JSON, markdown) never exercise the audio path. The `''` was almost certainly meant to catch *extension-less* files (`Makefile`); `startsWith` turned it into a universal claim. **Fix:** exact extension equality; explicit no-extension handling; order Audio/Image/Office before Text; BinaryReader as final fallback; a 10-line routing test (none exists — this is precisely the untested seam).

### 2. KG-03 — Every Ollama sampling/determinism knob is silently ignored (high; fix ≈ 3 lines)

`OllamaService.ts:62-64`: `temperature`, `repeat_penalty`, `seed` are commented out of the request options. The README documents them, the schema defaults `temperature: 0.1`, the benchmark harness sets `temperature: 0` (`scripts/benchmark.ts:46`) — none of it reaches the model. The default local path extracts at Ollama's own default (~0.8), which directly undermines the "Zero Hallucination" posture, and every local A/B (v4.5-vs-v5, canon arms) ran with uncontrolled sampling noise. Two cautions when re-enabling: `repeatPenalty: 0.3` is a broken default (Ollama semantics: <1.0 *promotes* repetition — the CLI help even claims the opposite), and re-running benchmarks after the fix may shift the leaderboard table in the README.

### 3. KG-02 — Failed chunks are checkpointed as done (high; fix ≈ 10 lines)

`generateKnowledgeGraph` catches everything and returns an empty graph (`KnowledgeGraphBuilder.ts:613-621`); `buildChunk` checkpoints whatever it gets (`:354`). A chunk that failed for transient reasons — rate limit, credits exhausted (the *exact* scenario `--resume` advertises), output truncation — is recorded as successfully-done-and-empty, and every future resumed run skips it. The user sees no difference between "this chunk had no extractable knowledge" and "this chunk was never successfully processed." Fix: make failure distinguishable (rethrow or sentinel), never append failed chunks, surface a `failed: N` count in the final summary. The same cache-the-failure anti-pattern exists in `CorpusAnalyzer` (KG-06): a failed glossary call is cached as an empty glossary and reused forever — observed in a real sidecar (`kg_tests/canon/baseline.json.corpus-profile.json`, 0 types / 0 predicates, while the same corpus's cloud profile has 18/14).

### 4. KG-04 — The merger destroys the cross-file edges v5 asks for (high; fix ≈ 15 lines + prompt edit)

Three artifacts collide: (1) `v5/user.hbs:33-43` instructs the model to *not* re-emit retrieved entities and to "just point relations at them by name"; (2) `v5/system.hbs:29` says relation endpoints "must each match a `name` in `entities` exactly" — a direct contradiction inside one rendered prompt; (3) `KnowledgeMerger.ts:389-404` drops any relation whose endpoint isn't an entity *of the same file* during the within-file pass, so a compliant cross-file edge never reaches the global merge. Today's self-run log shows the scale: up to **69 relations dropped from a single file** (`kg_tests/self/app.log`). Caveat, honestly: some of those drops are genuine model errors (named endpoints never emitted), and I did not partition the 69 into "retrieval-compliant" vs "hallucinated" — but the mechanism guarantees that *every* compliant cross-file edge is destroyed, at debug-level visibility. Fix: pick one contract (allow referencing retrieved names), validate within-file relations against (file entities ∪ retrieved names ∪ prior-graph names), and only enforce final referential integrity at the global stage, with an info-level drop count. `kg-gen metrics`' `danglingEndpointCount` will then actually measure something.

### 5. KG-06 — The authoritative glossary is never validated (high; fix ≈ 40 lines)

Under v5, the corpus glossary *is* the schema: its types/predicates become the enforced closed enums (`KnowledgeGraphBuilder.ts:377-406`) and the "use these and only these" system-prompt vocabulary. But `CorpusAnalyzer` does nothing to the LLM's output except trim (`dedupe`, `:233-235`). The real artifact from the self-run shows what gets through: 29 entity types (its own prompt says 8–20), `Programming Language`-style casing that collides with base `concept` → both `Concept`(99) and `concept`(58) in the final graph; 67 predicates including spaced phrases (`is a`, `part of` — duplicating base `part_of`) and the `has format`/`has length`/`has total chunks` attribute family that `EXAMPLE_STYLE_GUIDE.md` rule 5 explicitly bans. Garbage in → garbage *enforced*. Fix: a `normalizeGlossary()` step — snake_case + lowercase, case-insensitive dedupe against base vocab, hard caps (20 types / 15 predicates), drop `has_*` attribute predicates — plus: never cache a failed/empty glossary, and include a corpus-content hash in the cache key (today only file *paths* are hashed, so edits keep stale glossaries — `CorpusAnalyzer.ts:147-160`).

**Honorable mention — KG-05** (vocab three-way incoherence under `--classifier`) is the same disease as KG-06 from the other direction and shares the fix shape: one structure generates prompt text, examples constraints, hints, and the Zod enum. And **KG-08**: the grounding seam was explicitly built for a stronger checker — `bespoke-minicheck` in the Ollama library is a drop-in, local-first upgrade (MiniCheck, EMNLP 2024).
