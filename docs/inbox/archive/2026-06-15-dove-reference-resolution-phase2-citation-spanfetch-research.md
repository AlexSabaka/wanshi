# Grounding Implementation Decisions for a Citation Span-Fetch Feature (TypeScript/Node Knowledge-Graph Tool)

**Bottom line:** Adopt GROBID (CRF-only, native arm64 Docker image) as a local bibliographic parser — it is the single dependency that solves the crux problem (Q1), because it is the only locally-runnable, offline tool that emits BOTH a clean parsed reference list AND explicit in-text-marker→bibliography-entry links with resolved DOIs/arXiv IDs, and it fits comfortably in well under 4GB on a 16GB M4. Regex over pdf2json output cannot reliably recover the marker→entry mapping and should be retained only as a cheap arXiv-ID/DOI fallback.

## TL;DR
- **Q1 (the crux):** In-text-marker→bibliography linking from pdf2json-shredded text is NOT reliably solvable with regex+heuristics. In the Tkaczyk et al. JCDL 2018 evaluation, rule/regex tools achieved only 0.22 recall on reference parsing versus 0.66 for ML tools (F1 0.33 vs 0.71), and the marker→entry mapping additionally needs layout information that a prose text splitter destroys. A dedicated layout-aware parser is effectively required.
- **Q2:** Adopt **GROBID**. Its lightweight CRF-only image (native arm64 since v0.8.1) needs roughly 3GB RAM for citation extraction / 4GB for full structuring — trivial on a 16GB M4 — runs fully offline as a localhost service, and produces TEI/JSON with `<ref type="bibr" target="#bN">` links (citation-context resolution F1 0.76–0.91). This makes Q2 a **hard prerequisite** for sub-step 2b (Q5 = yes).
- **Q3:** Ship **id-bearing references in v1, but add a Crossref title→DOI resolver early** — Crossref's bibliographic matching reaches 96.3% F1 on real reference strings, so chasing id-less refs is worth it. Chain arXiv → Unpaywall(DOI) → Crossref/Semantic Scholar/OpenAlex(title), but expect only a minority of references to chain all the way to open-access full text (literature-wide OA share ≈ 27.9%).

## Key Findings

### Q1 — Marker→bibliography linking is the gating problem, and regex alone is insufficient
The task couples two sub-problems: (1) segment the reference section into discrete structured entries, and (2) resolve each in-text callout — numeric ("[23]", "[12,15]") or author-year ("(Smith et al., 2020)") — to the correct entry. pdf2json emits text runs in reading order; a prose-oriented splitter then fragments the dense, multi-column reference section into word-fragments, destroying the line/column/ordinal structure that both sub-problems depend on.

For numeric styles, resolution is mechanically simple *if* the list is correctly segmented and numbered (match bracket number to entry ordinal). For author-year, it requires fuzzy matching surname(s)+year against parsed entries while handling "et al.", letter-suffix disambiguation ("1996c"), and range/list expansion.

The accuracy gap between naive and ML/layout-aware approaches is large and well-documented:
- **Tkaczyk, Collins, Sheridan & Beel, JCDL 2018** ("Machine Learning vs. Rules and Out-of-the-Box vs. Retrained," arXiv:1802.01168), evaluating 10 parsers: ML-based and rule/regex tools had similar precision (0.77 vs 0.76), but *"applying machine learning-based tools results in a recall three times higher than in the case of non-ML-based tools (0.66 vs. 0.22)"* — i.e. ML F1 ≈ 0.71 vs rule/regex F1 ≈ 0.33. Concretely: *"The out-of-the-box tool achieving the best F1 is GROBID with F1 of 0.89, followed by CERMINE (F1 0.83) and ParsCit (F1 0.75). The tools with the worst F1 are: Citation-Parser (F1 0.27), Citation (F1 0.32) and PDFSSA4MET (F1 0.32)"* — the worst three are precisely the rule/regex tools.
- **GROBID's own benchmark** reports citation-context resolution — *"both the correct identification of the citation callout and its correct association with a full bibliographical reference"* — at **F1 between 0.76 and 0.91** depending on corpus. This is exactly the marker→entry link the feature needs; no lightweight regex approach comes close.

**Verdict:** Regex over pdf2json output is sufficient only to extract arXiv IDs/DOIs that appear verbatim in the text. It cannot recover the general marker→entry mapping. A layout-aware parser is required.

### Q2 — Bib-parser tooling decision: GROBID is the recommended dependency

**GROBID (Java/Docker; Apache 2.0) — STRONGLY RECOMMENDED.**
- *Extracts:* full-text TEI with header metadata, a clean parsed reference list, AND in-text citation markers linked to bibliography entries via `<ref type="bibr" target="#bN">` with character offsets and PDF coordinates, plus resolved DOI/PMID/arXiv IDs. The Python/Node client `--json` mode flattens this to JSON where each paragraph carries a `refs` array with `type`, `target`, `text`, `offset_start`, `offset_end` — directly usable for span selection.
- *Accuracy:* reference extraction/parsing ≈ 0.87 F1 on an independent PubMed Central set (90,125 references) and ≈ 0.90 on bioRxiv using the DL citation model; citation-context resolution 0.76–0.91 F1. Best out-of-the-box parser in multiple independent benchmarks.
- *Hardware footprint:* per GROBID docs, header extraction needs <2GB RAM, citation extraction ~3GB, full PDF structuring ~4GB (multithreaded). The CRF-only "lightweight" image is CPU-only and is the documented recommendation for low-resource systems. A **native arm64 image has existed since v0.8.1** (`lfoppiano/grobid`), tested on macOS. The "full" DL image (~8GB image + ~5GB embeddings) expects a GPU; CPU DL inference is ~50× slower.
- *Hardware-fit verdict (16GB M4):* **COMFORTABLE** with the CRF-only arm64 image. Avoid the full DL image (no CUDA GPU on Apple Silicon; the M4 GPU is not used by GROBID's TensorFlow/DeLFT stack).
- *Invocation:* local HTTP service (Dockerized) or Java library; the Node app calls it over localhost. Fully offline once models are downloaded. (Not a one-shot `spawn` subprocess by default, but self-contained and local.) Maintenance: actively developed (v0.9.x in 2025), Inria-backed, in production at ResearchGate, Internet Archive, CORE, CERN, Mendeley.

**anystyle (Ruby; BSD) — useful fallback, not a complete solution.**
Best-in-class reference-*string* parser with a tiny footprint (~2MB dictionary). In the Cioffi & Peroni (2022) benchmark it had the best overall reference-parsing scores (references F1 0.77; metadata F1 0.95; content F1 0.89). BUT its full-text "Finder" is explicitly experimental (trained on a single university's theses), and it does **not** link in-text markers to entries. Good CLI/subprocess parsing fallback; insufficient alone.

**refextract (CERN/Inspire, Python; GPLv2) — workable but HEP-tuned, no marker linking.**
Extracts references with DOIs, line markers and journal metadata; excels at DOIs. Lightweight. But it is High-Energy-Physics-tuned, does not link in-text callouts to entries, and the maintainers state it *"cannot be locally installed on a mac"* due to `mmap` resize usage — it must run in Docker. Latest release Oct 2025, so maintained.

**CERMINE (Java; AGPL v3) — good parser, licensing caveat, no marker linking.**
Reference parsing F1 ≈ 0.83 out-of-box (retrainable to ~0.92), NLM/JATS output, modest footprint, usable as a Java library or CLI. AGPL-v3 is a licensing consideration for a distributed tool. Focuses on reference extraction, not in-text marker linking.

**Science Parse / SPv2 (AllenAI; Apache 2.0) — NOT RECOMMENDED.**
Effectively unmaintained — the hosted service is shut down (*"This service is no longer maintained"*). SPv2 needs ~14GB RAM and *"will only run on a Linux host because of some optimizations in the tensorflow library."* Poor fit for a 16GB M4.

**Scholarcy — EXCLUDED.** Commercial/cloud-oriented; conflicts with offline-first.

**Pure-Node/JS/WASM — none viable for the parsing intelligence.** No mature pure-JS reference parser with in-text-marker linking exists. JS clients exist for *consuming* GROBID (HTTP) and the resolver APIs (e.g. `openalex-js`), but the parsing/linking model itself is not available natively in JS/WASM.

### Q3 — id→open-access resolution breadth
- **Trivial id paths:** arXiv ID → `arxiv.org/pdf/<id>`; DOI → Unpaywall `best_oa_location` (`url_for_pdf`); PMID → PMC.
- **Unpaywall:** free API, requires an `email` parameter, covers 30M+ articles. `best_oa_location` ranks by version (published > accepted > submitted), direct-PDF availability, and repository ranking (PMC and arXiv ranked above others).
- **Title→identifier resolvers (for the id-less majority):**
  - **Crossref** `query.bibliographic` — the DOI authority; no key required; "polite pool" via `mailto`. Historically ~50 req/s, but **rate limits are being revised from 1 Dec 2025** for the public/polite pools. Searches titles/authors/metadata, not full text.
  - **Semantic Scholar** — strong CS/ML and biomedical coverage; unauthenticated pool shared at 1000 req/s (throttled under load) or ~100 req/5min in practice; an API key grants a dedicated **1 req/s** starter limit. Has a dedicated `paper_title_search` (exact-title match) endpoint.
  - **OpenAlex** — 250M+ works, broadest coverage; 100,000 req/day and 10 req/s, polite pool via `mailto`. **Note: OpenAlex announced API keys will be required starting Feb 13, 2026, removing the email/polite-pool mechanism** — plan for a key.
- **Title-match accuracy:** Crossref's search-based bibliographic matching is the only source with rigorous numbers. Per Crossref Labs (Tkaczyk, *"Reference matching: for real this time"*): on 2,000 real unstructured reference strings, *"the best variant of search-based matching outperforms the legacy approach in F1 (96.3% vs. 92.5%) … precision … (98.09% …), and … recall … (94.56% …)."* No rigorous head-to-head precision/F1 benchmark of Crossref vs Semantic Scholar vs OpenAlex for free-text matching exists; the practitioner consensus is to **cascade** them (Crossref as DOI authority; OpenAlex broadest; Semantic Scholar best for CS/ML), accepting a match above a similarity threshold (community tools use ~85% title similarity, weighting title 0.6 / authors 0.2 / year 0.2).
- **Coverage reality (two hops):** A large share of references lack a deposited DOI — the arXiv:2511.18408 OpenCitations matching study found ~698M of ~1.81B Crossref references (~38.5%) have no DOI (Crossref's own blog frames it as *"43% of open/limited references … have no publisher-asserted DOI and no unstructured string"*). A high-precision heuristic matcher recovers only **58.31% of no-DOI references (69.45% including DOI-based queries), both at "100% precision, indicating that no incorrect matches occurred"** (arXiv:2511.18408). And once resolved to a DOI, only a minority of articles have a free full text: Piwowar et al. 2018 (PeerJ 6:e4375) found *"OA accounts for 27.9% of the scholarly literature as a whole (36.1% for the Web of Science sample)."*
- **Net:** Only a minority of a paper's references will chain all the way to open-access full text. **id-bearing-only is a defensible v1**, but a Crossref title resolver is high-ROI (≈96% F1) and should follow quickly.

### Q4 — Citation faithfulness / NLI (brief, per instructions)
- **Model:** MiniCheck is a strong, cheap, locally-runnable choice. Per Tang, Laban & Durrett (EMNLP 2024, arXiv:2404.10774): *"Our best system MiniCheck-FT5 (770M parameters) outperforms all systems of comparable size and reaches GPT-4 accuracy"* at *"400x lower cost,"* and it is robust without per-dataset threshold tuning (its accuracy drops only 0.4% without threshold tuning, versus 9.2% for SummaC-CV).
- **(a) What the "claim" should be:** The raw citing sentence ("citance") is frequently *not* a clean standalone claim — it carries anaphora, bundles multiple facts, and includes citation markers and proper names that add noise. Scientific-claim-verification work (SciFact built its claims by *rewriting citances into atomic claims*) and modern fact-checking pipelines decompose/decontextualize the citing sentence into atomic, standalone claims before NLI. An LLM-reformulated "what this citation is used to support" generally outperforms verbatim input, at the cost of extra compute and an added hallucination surface — a worthwhile later A/B.
- **(b) Uncertain/neutral band:** Specialized NLI/fact-checkers emit a continuous support score; calibrate **two thresholds** to carve out a neutral/abstain band rather than forcing a binary verdict. This mirrors the explicit **NEI ("Not Enough Info")** class in SciFact/SciFact-Open. Orienting literature for a later deep dive: CiteWorth (cite-worthiness, 1.1M sentences across 10 domains), SciFact / SciFact-Open, MultiVerS (weak supervision + full-document context), and recent citation-hallucination benchmarks (e.g., CiteAudit). Note a measurement caveat raised in recent work (arXiv:2606.09376): reference-free faithfulness metrics measure precision only and can reward abstention — keep an eye on coverage, not just "are stated claims supported."

### Q5 — Sequencing sanity-check
**Yes — 2b is not reliably attemptable without first solving Q2.** Clean in-text-marker→reference linking requires layout-aware parsing that pdf2json + a prose splitter structurally cannot provide (the evidence in Q1: regex/rule recall ≈ 0.22 and no marker→entry resolution; GROBID citation-context F1 0.76–0.91). Therefore adopting GROBID (Q2) is a **hard prerequisite** for sub-step 2b, not an independent choice. The planned order — 2a (fetch OA full text + handle PDFs) → 2b (marker→entry linking + span selection) → 2c (NLI labeling) → 2d (title→id resolver) — is sound, with the explicit dependency that **2b must be gated on standing up GROBID**.

## Details
This report synthesizes primary documentation, GitHub repositories, and peer-reviewed/benchmark literature: GROBID docs and repo (RAM figures, arm64 image, TEI `<ref target>` output, benchmark F1s); the Tkaczyk et al. JCDL 2018 ten-parser evaluation (regex vs ML recall gap; per-tool F1s); the Cioffi & Peroni 2022 tool comparison (anystyle/CERMINE/refextract/Science Parse scores); Crossref, OpenAlex, Semantic Scholar, and Unpaywall API documentation and rate-limit notices; the Crossref Labs reference-matching blog series (Tkaczyk); arXiv:2511.18408 (no-DOI reference matching coverage); Piwowar et al. 2018 (OA share); the MiniCheck paper (Tang/Laban/Durrett, EMNLP 2024); and SciFact / CiteWorth / MultiVerS for citation verification framing.

Architecturally, the recommended pipeline is: (1) run GROBID locally (CRF-only arm64 image) on the citing PDF to obtain TEI/JSON with marker→entry links and resolved IDs; (2) for each evidence-bearing citation, resolve the target entry's identifier via arXiv → Unpaywall(DOI) → Crossref(title) → Semantic Scholar/OpenAlex(title); (3) fetch the cited OA PDF and run GROBID on it too, to get clean body text and offsets for span selection; (4) decompose the citing sentence into atomic claims and run MiniCheck against retrieved spans, applying a two-threshold support/neutral/unsupported labeling.

## Recommendations
1. **Stand up GROBID (CRF-only arm64 Docker image) as a local service first** — this unblocks 2b. Budget ~4GB RAM (trivial on 16GB). Use `lfoppiano/grobid` arm64 tag; do not pull the full DL image.
2. **Use GROBID's TEI/JSON `<ref target>` links as the primary marker→entry mapping**, with character offsets for span selection. Keep a regex arXiv-ID/DOI extractor as a cheap fallback and cross-check, not as the primary mechanism.
3. **Implement OA resolution as a cascade: arXiv → Unpaywall(DOI) → Crossref(title) → Semantic Scholar/OpenAlex(title).** Ship id-bearing references in v1; add the Crossref title resolver in v1.1 (high ROI at ≈96% F1). Always send a `mailto`; provision an OpenAlex API key ahead of the Feb 13, 2026 requirement and account for the new Crossref limits from 1 Dec 2025. Accept title matches only above a tuned similarity threshold (~0.85 title similarity, with author/year corroboration).
4. **For NLI, run MiniCheck locally; decompose citances into atomic claims; calibrate a two-threshold neutral/abstain band** (mirroring SciFact's NEI class) rather than forcing binary verdicts. Treat LLM claim-reformulation as an opt-in enhancement to A/B against verbatim sentences.
5. **Set explicit benchmark triggers.** If GROBID's marker→entry links measure below ~0.80 F1 on a held-out sample of your real CS/ML arXiv corpus, escalate to the DL citation model (more RAM/time) or add anystyle as a parsing fallback. If id-less title-resolution precision on a manual audit falls below ~0.95, raise the similarity threshold or restrict to id-bearing references.

## Caveats
- Crossref's matching F1 figures are self-published (Tkaczyk / Crossref Labs blog) — methodologically transparent and widely cited, but not peer-reviewed.
- No rigorous head-to-head Crossref vs Semantic Scholar vs OpenAlex free-text-matching benchmark exists; cascading is the pragmatic consensus, and all three use token/keyword search (not semantic embeddings), so they miss references with misspelled distinctive words.
- GROBID's benchmark F1s derive from PMC/bioRxiv corpora; CS/ML arXiv PDFs (heavy math, varied templates, two-column layouts) may differ, and end-to-end numbers fold in PDF-extraction noise — validate on your own corpus.
- OA-coverage figures vary by year and field; the 27.9% literature-wide / 36.1% WoS-sample numbers (Piwowar 2018) are the defensible large-scale baseline, and OA has risen since, but vendor/library "50–85%" figures are soft and should be discounted.
- The "no-DOI reference" share (≈38.5% per arXiv:2511.18408; "43% with no DOI and no string" per Crossref's blog) and the 58–69% recovery rate indicate that many references will simply not be resolvable to OA full text — design the feature to degrade gracefully (emit an unverified citation edge rather than failing).
